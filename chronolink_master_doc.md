ChronoLink: Master Architecture & Product Requirement Document (PRD)1. Product SummaryChronoLink is an enterprise-grade, deterministic academic scheduling and resource optimization engine built specifically for Symbiosis Skills and Professional University (SSPU). It bridges complex CP-SAT mathematics (Google OR-Tools) with an intuitive, high-density React frontend. ChronoLink acts as a "logic machine" that translates university constraints, physical resource availability, and faculty preferences into a 100% clash-free timetable. It replaces manual, error-prone scheduling with deterministic algorithmic output wrapped in a high-density, dark-mode native SaaS interface inspired by Vercel and Linear.2. Goals and Non-GoalsGoals:Generate mathematically verified, clash-free schedules using a CP-SAT solver.Provide a high-density, "Excel-like" strict CSS Grid UI for maximum data visibility.Eradicate silent failures by providing deterministic visual feedback ("Conflict Glow" / Unsat Core) when schedules are infeasible.Support role-based, filtered views so constituents (Students/Faculty) only see their relevant data.Enable rapid enterprise onboarding via bulk CSV data importing.Non-Goals:AI-driven "heuristic" or generative guessing for schedules (must remain strictly deterministic).Support for messy, flexible, or standard HTML <table>/Flexbox-based master grid UI.Browser-based persistent storage (localStorage must NOT be used as the source of truth).Native mobile app development (focus is on a responsive, zoom-resilient web grid).3. User StoriesAdmin (University Planner):As an Admin, I want to define dynamic grid horizons (total_days, total_slots_per_day) so I am not locked into a 5-day/10-hour week.As an Admin, I want to drag-and-drop a CSV to bulk-import Rooms, Faculty, and Subjects so I avoid manual data entry.As an Admin, I want to use a "Constraint Builder" with cascading dropdowns to set rules (e.g., Location Preferences, Max Sessions) without writing raw JSON.As an Admin, if the solver fails, I want to see the exact conflicting rules pulse red ("Conflict Glow") so I can resolve the deadlock.As an Admin, I want to click "Go Live" to change a schedule status to PUBLISHED so that Students and Faculty can view it.As an Admin, I want to use the AdminDirectory to view and filter all faculty and student groups by tags (e.g., Semester 1).Faculty:As Faculty, I want to set my availability (unavailable slots) via a visual, reactive grid before the schedule is generated.As Faculty, I want to view my published schedule in a clean grid showing only my assigned classes.Student:As a Student, I want to log in and see a read-only, zoom-resilient grid of my specific cohort's weekly timetable.5. End-to-End User FlowsFlow 1: Schedule Generation & Resolution (Admin)Admin navigates to /dashboard (TimetableWorkspace.tsx).Admin sets grid config (total_days=5, total_slots_per_day=20).Admin adds entities (Faculty, Rooms, Subjects) manually or via BulkDataImporter.tsx.Admin creates rules in ConstraintBuilder.tsx (e.g., Target: Dr. Strange, Rule: Unavailable Monday 09:00).Admin clicks "Run Solver".Variant A (Success): Backend returns 200 OK. Grid populates with ScheduledSession cards.Variant B (Infeasible): Backend returns 422. Frontend extracts conflicting_constraint_ids. Offending rule cards pulse red (ring-destructive). Admin modifies rules and re-runs.Admin clicks "Go Live". Backend updates schedule status to PUBLISHED.Flow 2: Viewing Personal Schedule (Student/Faculty)Student logs in.App routes to /student (StudentTimetablePage.tsx).Frontend calls GET /schedules/published/student/{group_id}.UI renders the strict CSS Grid populated only with sessions matching student_group_id.6. UI/UX SpecificationGlobal UI Principles:Theme: Dark-mode native (Vercel/Linear aesthetic).Colors & Styling: Strict Tailwind classes (bg-background, text-muted-foreground, rounded-md).Banned Practices: No inline styles (style={{ background: 'red' }}). No rounded-2xl or rounded-xl.Constraint Builder Inputs: Must use tactile sliders, reactive grids, and cascading dropdowns. NO raw JSON inputs.Grid Standard (TimetableGrid.tsx & Page Grids):Architecture: Strict CSS Grid. Banned: Flexbox or HTML <table> for layout.Columns: grid-template-columns: 80px repeat(total_days, 1fr).Rows: grid-template-rows: 40px repeat(total_slots_per_day, minmax(38px, 45px)).Mapping: ScheduledSession must map via inline style/Tailwind arbitrary value: gridColumn: session.day_of_week + 2, gridRow: session.start_slot + 2, gridRowEnd: span session.duration.Card Density: p-1.5, text-xs (12px) for titles, text-[10px] for meta-data. m-[1px] margin to prevent border overlap.Conflict Glow (Error State):Triggered on 422 HTTP response. Applies ring-2 ring-destructive bg-destructive/10 animate-pulse border-red-500 to broken rule cards.8. Data Model and Domain LogicEntities & Fields:Teacher: id (UUIDv4), name, email, tags (Array of Strings/Relational).Room: id (UUIDv4), name, capacity, building_tag (String).StudentGroup: id (UUIDv4), name, tags (Array of Strings).Subject: id (UUIDv4), name.Constraint: id (UUIDv4), target_type (Enum), target_id (UUIDv4), rule_type (Enum), value (JSON/Dict), weight (Int).Schedule: id (UUIDv4), status (Enum: DRAFT, PUBLISHED).ScheduledSession: id, schedule_id, teacher_id, room_id, student_group_id, subject_id, day_of_week (Int), start_slot (Int), duration (Int).Derived Logic:Implicit Location Mapping: If StudentGroup has a constraint preferred_building: "Block_A", solver applies a soft constraint (weight/reward) to place sessions in Room where building_tag == "Block_A".9. APIs and IntegrationsEndpoints:POST /solve: Payload includes total_slots_per_day, total_days, arrays of all live entities, and constraints. Returns 200 (Sessions) or 422 (Infeasible with conflicting_constraint_ids).POST /schedules/{id}/publish: Changes state to PUBLISHED.GET /schedules/published/student/{group_id}: Returns filtered active sessions for a student.GET /schedules/published/teacher/{teacher_id}: Returns filtered active sessions for faculty.GET /teachers, GET /groups: Fetch registries for AdminDirectory.ZeroTrustAuditMiddleware:Intercepts requests, validates logic (prevents Room/Teacher/Group physical overlaps), generates SHA256 digest of canonical session rows, appends Verification Secured:{digest} header.10. Customization and ConfigurationDynamic Horizons: total_days and total_slots_per_day are globally configurable by the Admin, dictating grid generation loops. Hardcoded arrays (DAYS, TIME_ROWS) are strictly prohibited.Tags: Admins configure tags (e.g., Semesters, Departments) applied to StudentGroups and Teachers to allow cascading dropdown filtering in the UI.11. Technical ArchitectureStack:Frontend: React, TypeScript, Tailwind CSS, shadcn/ui, uuid npm package.Backend: FastAPI (Python), SQLModel.Solver Engine: Google OR-Tools (CP-SAT) in solver_engine.py.State Management: Centralized useState/useReducer in TimetableWorkspace.tsx. localStorage is banned for core scheduling data.Data Flow Patterns:Modals generate uuidv4(), append to global React state. React state builds payload for /solve. Axios hits FastAPI. FastAPI feeds OR-Tools. Result populates DB/Frontend.12. Security, Privacy, ComplianceZeroTrustAuditMiddleware: Acts as a fail-safe validation layer against overlapping allocations.Role-Based Access Control (RBAC): UI routing and API filtering strictly map to AppRole (developer, admin, faculty, student). Student/Faculty MUST NOT be able to query the master grid directly.Data Protection: localStorage is explicitly forbidden to prevent cache poisoning or unauthorized reading of enterprise timetable drafts.13. Observability and OperationsLogging: FastAPI standard logging.Audit: Middleware returns verification hashes for cross-checking logical output integrity.UI Diagnostics: The "Conflict Glow" acts as visual diagnostic tracing for mathematically impossible inputs.14. Performance and Reliability TargetsBottlenecks: CP-SAT solving can be CPU/RAM intensive.Mitigation:Utilize implicit mappings (building tags) as soft constraints (model.Maximize(10)) rather than hard constraints to speed up the solver matrix.Development optimization: Add .cursorignore / .antigravityignore to prevent IDE memory leaks from massive file indexing.15. Testing StrategyUnit Testing: Verify coordinate mapping math in TimetableGrid.tsx (e.g., dayIndex(day) + 2).Integration/Solver Testing: Create an "Impossible Teacher" scenario (mark unavailable, force into same slot). Verify API returns 422 and UI triggers Conflict Glow.Grid Integrity Testing: Ensure CSS grid rows maintain exactly 38px/45px height regardless of session density.16. Release and Migration PlanPurge Legacy: Prior to deployment, execute a full script sweep to eradicate remaining "Stitch Legacy" code, inline styles, and localStorage references.Rollout: Deploy Admin Workspace first. Validate CP-SAT outputs. Deploy Student/Faculty read-only dashboards sequentially.17.
